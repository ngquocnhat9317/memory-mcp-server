## 1. Context and how this spec was assembled

Wave 3 (v1.3.0) is fully shipped and published: WI-1, WI-2, WI-3, WI-5, WI-8, WI-9 are all confirmed in code; WI-4 is done except for a GIF/asciinema asset that requires the owner to record a real terminal session (not an engineering task, excluded from this spec). WI-6, WI-6b, WI-7 are Wave-4 candidates explicitly gated behind a 4–6 week real-usage evidence window (main spec §9.3) that cannot possibly be open yet — excluded here.

Draft 0.1 of this spec carried five work items. A code-level review of that draft against `develop @ d8fdb60` found a structural fact that eliminated two of them (see HT-12a in §3 and the full rationale in §8): both the proposed time-decay tie-break (WI-12) and the proposed importance nudge (WI-14) act on the `importance` position of the ranking comparators — a position that only ever executes when two candidates' BM25 float scores tie **exactly**, which effectively never happens on real data. Both features would have passed their acceptance tests (fixtures can construct exact ties) while delivering no observable change to real users — the precise "unmet expectations" failure mode this project's own review process exists to catch. Per owner decision (2026-07-12), items whose practical effect is near zero or whose value lacks evidence do not ship in this release; both are recorded in §8 rather than silently dropped, and their underlying ideas fold into the v1.4.0 blended-scoring work where the comparator is redesigned as a whole, behind the existing evidence gate.

What remains is deliberately narrow: three items that are either pure consistency (WI-10), pure measurement infrastructure (WI-11), or have an immediate, directly observable consumer at the moment they act (WI-13). Notably, **this release now introduces zero new configuration variables** — stricter than draft 0.1, and fully aligned with the zero-config philosophy.

### 1.1. Evidence this spec is built on (not re-derived — pulled from existing memory records)

- **Root-cause tool-adoption finding** (`mem_6f05457c`, session `sess_33d55edb`): optional fields the agent must proactively fill in, consumed only by a report/audit layer nobody reads, get ~0% adoption regardless of documentation quality — proven on this exact repo's own telemetry. **Consequence for this spec:** every WI below is either fully server-side (zero new agent effort) or purely advisory and consumed at the moment of writing. This is also why a "step-level provenance" idea was rejected outright (§8), and why WI-13's duplicate hint must stay high-precision (a noisy hint gets ignored, then it is dead — same pattern).
- **Value-lever positioning** (`mem_737cef1c`, session `sess_eaa86f9c`): perceived value = useful recalls / (effort + noise); levers in priority order are (a) time-to-first-value, (b) recall ranking quality, (c) trust via provenance, (d) corpus quality.
- **Growth-stage context** (`mem_c1711a0d`, session `sess_46979050`): the product has ~0 external users; distribution/positioning work outranks engineering investment in absolute terms. This spec stays deliberately small (3 WIs, no new dependencies, no new configuration, no schema change) in acknowledgment of that.
- **WI-8/WI-9 implementation record** (`mem_6e778fe1`, session `sess_8ed162a0`): confirms the recall candidate pool cap (`RECALL_CANDIDATE_POOL = 50`), the current tie-break order, and that a "blended scoring mechanism" was explicitly reserved for v1.4.0 pending real-use evaluation — the decision that §8's two dropped items now defer to.

## 2. Goals and non-goals

### 2.1. Goals

- **G1 — Provenance is consistent everywhere a memory is read**, not just in auto-recall (closes WI-2's deliberately deferred scope).
- **G2 — Future ranking changes are validated, not guessed.** A reusable eval base exists before any further ranking tuning is attempted (unlocks OQ-1 and doubles as the Wave-4 evidence-gate G-d precondition).
- **G3 — The memory store actively resists degrading itself.** A near-duplicate is flagged at the moment of creation, when the signal is cheapest to act on — directly serving lever (d), corpus quality — and the flag is precise enough that agents keep trusting it.

### 2.2. Non-goals

- **No ranking-order changes of any kind in this release.** Neither auto-recall nor `memory_search` ordering changes — the two draft-0.1 items that would have touched ranking (time decay, importance nudge) are dropped per §8. The next ranking change happens, if ever, as the v1.4.0 blended-scoring work behind the evidence gate.
- **No new dependencies, no vector/semantic search, no network calls** — unchanged discipline from Wave 3.
- **No new configuration variables at all** (tightened from draft 0.1's "no new *required* configuration").
- **No tool-surface changes** — no tools added or removed; only enrichment of existing tool outputs.
- **No schema/migration changes.**
- **No step-level reasoning provenance, no reasoning graph, no server-side AI summarization** — considered and rejected; see §8.

## 4. Work items

### WI-10 — Provenance consistency: surface `source` on every memory read path (P1)

**Answer to the filtering question:** makes every recall — not just auto-recall — **more trustworthy** (G1, lever c).

**Description.** `recallRelatedMemories` already turns a `memories.metadata` blob into a clean `source{session_id, session_title, created_at}` field (WI-2, shipped). `memory_search`, `memory_get`, `memory_list`, and the return values of `memory_save`/`memory_update` all go through the same `rowToRecord` function (`src/tools/memory.ts:41-53`) and currently expose only the raw `metadata` object, forcing every other caller to know the internal `source_session_id`/`session_title` key names. This is the same data, presented inconsistently.

**Design.**
1. Add a shared helper to `src/utils.ts`:
   ```ts
   export interface MemorySource {
     session_id: string;
     session_title: string;
     created_at: string;
   }

   export function extractMemorySource(
     metadata: Record<string, unknown> | null,
     createdAt: string
   ): MemorySource | undefined {
     const sessionId = metadata?.source_session_id;
     const sessionTitle = metadata?.session_title;
     if (typeof sessionId === "string" && typeof sessionTitle === "string") {
       return { session_id: sessionId, session_title: sessionTitle, created_at: createdAt };
     }
     return undefined;
   }
   ```
2. Add optional `source?: MemorySource` to `MemoryRecord` (`src/types.ts:8-18`).
3. `rowToRecord` (`src/tools/memory.ts:41-53`) calls `extractMemorySource(parseJsonObject(row.metadata), row.created_at)` and attaches the result. Because `rowToRecord` is shared, this enriches `memory_search`, `memory_get`, `memory_list`, and the record echoed back by `memory_save`/`memory_update` — a deliberate scope decision (see note below), not scope creep: it is one call site, zero new queries, purely additive.
4. Refactor `recallRelatedMemories` (`src/tools/reasoning.ts:327-346`) to call the same helper instead of its inline duplicate — pure refactor, must not change existing WI-2 test outcomes.

**Scope note:** the original OQ-3 framing only asked about `memory_search`. Because the extraction lives in the one shared `rowToRecord` function, restricting it to `memory_search` alone would cost *more* code (a second code path) than enriching all read paths uniformly. Resolved as OQ-7: proceed uniformly (owner sign-off with this revision).

**Functional requirements:**
- FR-10.1: `MemoryRecord` gains optional `source?: MemorySource`; no other field changes shape.
- FR-10.2: `rowToRecord` populates `source` whenever `metadata` carries both `source_session_id` and `session_title`; the field is omitted (not `null`) otherwise.
- FR-10.3: `recallRelatedMemories` uses the same `extractMemorySource` helper; its own output shape and existing tests are unchanged.
- FR-10.4: No schema/migration change — purely derived from existing `metadata`/`created_at` columns.
- FR-10.5: `memory_search`'s tool description gains one sentence documenting the `source` field, mirroring the pattern already used in `reasoning_start_session`'s description.

**Acceptance criteria:**
- AC-10.1: A memory persisted via `reasoning_complete_session(save_as_memory=true)`, later found via `memory_search`, has `source.session_title` equal to the originating session's title.
- AC-10.2: A memory created via plain `memory_save` → `memory_search`/`memory_get` results have no `source` field.
- AC-10.3: Existing WI-2 acceptance tests (auto-recall provenance) pass unmodified after the refactor.

**Files expected to change:** `src/utils.ts`, `src/types.ts`, `src/tools/memory.ts`, `src/tools/reasoning.ts`, `src/__tests__/memory-tools.test.ts`, `src/__tests__/wave3-value-loop.test.ts`. Effort: **S** (half a day).

### WI-11 — Recall relevance eval base (P1 — foundational, build first)

**Answer to the filtering question:** makes every *future* ranking change measurable instead of guessed (G2) — and is a direct precondition for both OQ-1 (BM25 column weighting) and the Wave-4 evidence gate's G-d condition, so it pays for itself twice. With WI-12/WI-14 dropped, this is also the artifact that will *prove or disprove* their underlying ideas when the v1.4.0 blended-scoring work is evaluated — the eval base outlives this release.

**Description.** No reusable, intentional set of recall test cases currently exists. Individual behaviors are covered ad hoc across `wave3-value-loop.test.ts` and `wave3b-recall-scoping.test.ts`, but there is nothing a future contributor can run before/after a ranking change (e.g. OQ-1's BM25 weighting, or the v1.4.0 blended score) to see what moved.

**Design.** Add `src/__tests__/fixtures/recall-eval-cases.ts` — an exported array of cases, each with: a `description`, a `title` (the recall query), a list of `seedMemories` (content/type/tags/importance/workspace, deliberately separated per the AC-1.1/AC-8.4 lesson so results aren't ambiguous), and an `expected` assertion (top id, or a full expected order). Minimum 10 cases:

1. Multi-term match beats single-shared-word match (WI-1/WI-8 core).
2. Same-workspace beats other-workspace at equal match count (WI-9).
3. Other-workspace with strictly more matched terms still beats same-workspace with fewer (WI-9 "soft, not hard").
4. Importance tie-break when matched/workspace/rank all tie (constructed exact-tie fixture — documents that this branch exists, and per HT-12a, that it is fixture-only territory in practice).
5. Recency (`updated_at`) tie-break when importance also ties.
6. One-word-overlap junk excluded when the title has ≥3 significant terms (the floor).
7. Short (1–2 word) title keeps a 1-term match (floor exemption).
8. No candidate clears the floor → exactly one best-ranked fallback returned.
9. A memory with a `source` field is recalled with `source` intact and correct (locks WI-10).
10. Duplicate-hint precision case, added together with WI-13: a save sharing topic vocabulary but below the dedupe floor produces **no** hint; a true near-duplicate produces one. The fixture file is meant to grow with the codebase, not be frozen at 10.

A companion `src/__tests__/recall-eval.test.ts` seeds a temp DB per case and drives the real registered MCP tools (`reasoning_start_session` for auto-recall cases, `memory_search` for search cases) — reusing the harness convention already proven in `wave3b-recall-scoping.test.ts` (temp DB via `mkdtemp`, `RegisteredToolMap`, real `register*Tools` calls). Workspace-dependent cases set `MEMORY_WORKSPACE` per case, which works because `getWorkspace()` reads the env at call time.

**Functional requirements:**
- FR-11.1: Fixture file contains ≥10 cases across the categories above, each with a documented, deliberately-separated setup.
- FR-11.2: A test suite runs every case against the live tool path and asserts the expected outcome.
- FR-11.3: The fixture is structured so a future ranking change (OQ-1, or any Wave-4 tuning) can be validated by re-running the same suite before/after, satisfying G-d.
- FR-11.4: No production code changes in this WI — test/fixture-only.

**Acceptance criteria:**
- AC-11.1: `npm test` passes with the new suite included; coverage does not regress below the current baseline (98.35% per the WI-8/WI-9 record).
- AC-11.2 (manual, documented in the PR description, not a permanent test): temporarily reverting a ranking criterion to prove the suite has teeth — at least 3 fixture cases must fail when e.g. the workspace tie-break is disabled.

**Files expected to change:** `src/__tests__/fixtures/recall-eval-cases.ts` (new), `src/__tests__/recall-eval.test.ts` (new). Effort: **S–M** (mostly authoring good fixtures).

### WI-13 — Duplicate-surfacing hint on `memory_save` (P2)

**Answer to the filtering question:** actively defends corpus quality (G3, lever d) instead of relying on the agent to notice duplication later during search. Unlike the two dropped items, its effect is **directly observable at the moment it fires** (a field in the save response, consumed immediately by the agent that is mid-save) — it does not depend on ranking-tie mechanics to matter.

**Description.** `memory_save` currently inserts unconditionally. A near-duplicate of an existing memory is only discoverable later, reactively. The matched-term-count machinery from WI-8 can detect this at save time — but **not with the recall floor as-is** (review finding MAJOR-2, now HT-13a): the recall floor (≥2 of ≤8 significant terms) answers "is this relevant?", and in any topically-focused store nearly every same-type pair shares 2 significant terms. Reusing it unchanged would fire the hint on almost every save, agents would learn to ignore the field, and the feature would die the exact noise-death documented in `mem_6f05457c`. Duplicate detection therefore gets its **own, much stricter floor**.

**Design.**
1. Extract the matched-term-count machinery currently inline in `recallRelatedMemories` (`src/tools/reasoning.ts:291-316`) into a shared helper in `src/utils.ts` (e.g. `countMatchedTerms(database, terms): Map<rowid, matched>`). `recallRelatedMemories` keeps its existing recall floor (`terms.length >= 3 ? 2 : 1`) — **unchanged behavior**, guarded by WI-11's fixture.
2. New dedupe-specific floor, a separate hardcoded constant with its own rationale (never shared with the recall floor):
   ```
   terms   = toRecallTerms(newContent)            // existing helper: ≤8 significant terms
   required = terms.length <= 4
                ? terms.length                     // short content: must match ALL terms
                : Math.max(4, Math.ceil(0.6 * terms.length))  // long content: ≥60%, min 4
   ```
   With the 8-term cap this means 5-of-8 for typical long content, 4-of-4 for short content — a candidate must share the *majority* of the new memory's vocabulary, not just its topic. **Known limitation (stated, accepted):** `toRecallTerms` takes the first 8 significant tokens, which biases toward content openings (often generic phrasing like "Decision:" / "Root-cause..."); the strict majority floor compensates by requiring most of even those tokens to co-occur. If real use shows recall/precision problems, revisit term sampling then — not speculatively now.
3. In the `memory_save` handler (`src/tools/memory.ts:269-304`), **before** inserting: run the check against existing memories restricted to the same `type` (hard filter — a `fact` and a `decision` about the same topic are not automatically duplicates). Cap results at `DUPLICATE_HINT_LIMIT = 3` (hardcoded — advisory infrastructure, not a user tunable; resolved as OQ-6), ordered by `matched DESC, rank ASC`.
4. Response placement (resolves review finding MINOR-5, per HT-13b): `structuredContent` stays the `MemoryRecord` object at the top level with `possible_duplicates` added as one more optional top-level key alongside it — i.e. `{ ...record, possible_duplicates?: [...] }`. The record's `id` remains at the top level, so the existing telemetry `buildEvent` (`structured.id`) is unaffected. The field appears **only when ≥1 candidate clears the dedupe floor** — omitted entirely otherwise, matching the "no noise by default" pattern (`source`, cold-start nudge). Shape: `Array<{id, type, snippet, matched_terms}>`. When present, append one sentence to the text response (mirrors the existing `recallNote` pattern): *"N possible duplicate(s) found — review before assuming this is new information."*
5. `memory_save`'s tool description gains one line: agents should review `possible_duplicates` and consider `memory_update` on the existing id instead of creating a near-duplicate. The save always succeeds regardless — advisory, never blocking.

**Functional requirements:**
- FR-13.1: `memory_save` runs the duplicate check against existing same-`type` memories using the dedupe floor in point 2 — a separate constant from the recall floor, which stays untouched.
- FR-13.2: `possible_duplicates` appears only when ≥1 candidate clears the dedupe floor, capped to 3, ordered by `matched_terms DESC, rank ASC`.
- FR-13.3: The save always succeeds regardless of duplicates found — non-blocking, informational only.
- FR-13.4: `possible_duplicates` lives at the top level of `structuredContent` next to the record fields; `structured.id` keeps working for telemetry (HT-13b).
- FR-13.5: `memory_save`'s tool description documents the field and the suggested agent action.
- FR-13.6: No match found → no field, no extra token cost.

**Acceptance criteria:**
- AC-13.1: Saving content that shares the majority of its significant terms (per the point-2 formula) with an existing same-`type` memory → response includes `possible_duplicates` containing that memory's id.
- AC-13.2: Saving content that shares only 2–3 topic words (above the old recall floor, below the dedupe floor) with an existing same-`type` memory → **no** `possible_duplicates` (precision lock — the case that kills the naive design; also WI-11 fixture case 10).
- AC-13.3: Saving a memory of a different `type` from an otherwise near-identical existing memory → no `possible_duplicates` (type is a hard filter).
- AC-13.4: Saving into an empty or unrelated store → no `possible_duplicates` field present at all.
- AC-13.5: The new memory is always created regardless of duplicates found.

**Files expected to change:** `src/utils.ts` (extracted shared helper + dedupe floor), `src/tools/reasoning.ts` (refactor to use shared helper), `src/tools/memory.ts` (duplicate check + response field + description), `src/__tests__/memory-tools.test.ts`, `src/__tests__/fixtures/recall-eval-cases.ts` (case 10). Effort: **S–M**.

**Risk note:** one extra FTS-backed check per `memory_save` call (the OR candidate query plus up-to-8 per-term queries — the same bounded pattern recall already runs per session start) — negligible on a local SQLite file, on a write path that is not hot. False positives are still possible in principle; the majority floor plus the same-`type` filter plus the advisory-only contract bound the damage.

### WI-15 — Memory-scope hygiene: keep agent persona out of the store (P2)

**Answer to the filtering question:** defends corpus quality (G3, lever d) at the *definition* layer — the store's own guidance currently permits, and in one clause actively legitimizes, content that is about the agent rather than about the project or user.

**Description.** A quality review of a live store (2026-07-16) found a persona memory saved by a visiting agent — `"Agent Identity: Hermes Agent, created by Nous Research. Helpful, knowledgeable, and direct."` — stored as `type=fact`, `importance=5`, no `agent_id`, alongside companion persona rows saved as `preference` ("Communication Preference: ..."). None of these were rule violations. They were *legitimate* saves under the current definitions, which is exactly the problem:

- `fact` is described only as `'fact' (durable info)` (`src/schemas/memory.ts:34-35`) — the only type description with no subject; and `fact` is the **default** type, so unclassified saves land there.
- `preference` is described as `'preference' (user/agent preference)` — the "agent" half invites persona boilerplate.
- The GUIDELINES "Do Not Store" list excludes content by *sensitivity or noise* (secrets, chain-of-thought, debug noise, raw dumps) but never by *subject* — nothing excludes agent self-description.
- The GUIDELINES Rule Of Thumb — store only what "will help a future agent or future run" — is satisfied, from the saving agent's perspective, by its own identity: it does help its own future runs. An agent can follow every rule and still pollute the store.

The practical recall risk is low since v1.3.0 (BM25 ranking plus the quality floor keep persona rows out of unrelated recalls), so this ships as guidance hardening on both agent-facing surfaces — **not** as a write-time validator (see the considered-and-rejected note below). Same dual-placement precedent as WI-9's tag-hygiene rule: state it once in the guide, once on the tool surface.

**Design.** Two layers, two distinct audiences:

1. **`GUIDELINES.md`** (reaches agents that fetch the guide via `get_usage_guide`):
   - Add a fifth "Do Not Store" bullet:

     ```
     - agent self-identity or persona boilerplate (who the agent is, who
       created it, its tone or communication style) — this store holds
       knowledge about the project, domain, and user, not about the agent
       itself
     ```

   - Reword the Rule Of Thumb closing line from "will help a future agent or future run" to **"will help a future task on this project or for this user"** — moving the criterion from *helps the agent* to *helps the work*, which closes the clause the persona save walked through.
   - Bump the guide's `Version:` line and update the `guide_version` assertion in `src/__tests__/reasoning-audit-tools.test.ts` in the same change (repo version-sync rule).
2. **Schema and tool descriptions** (reach every MCP client at tool-load time, including agents that never fetch the guide — the failure path actually observed):
   - `type` description in `src/schemas/memory.ts`: `'fact' (durable info)` → `'fact' (durable info about the project, domain, or user — not about the agent itself)`; `'preference' (user/agent preference)` → `'preference' (user preference)`.
   - `memory_save` tool description (`src/tools/memory.ts:233`): append one sentence — *"Store knowledge about the project, domain, or user — not the agent's own identity or persona."*

**Functional requirements:**
- FR-15.1: GUIDELINES gains the persona "Do Not Store" bullet and the reworded Rule Of Thumb; its `Version:` line is bumped and the `guide_version` test assertion is synced in the same change.
- FR-15.2: The `fact` and `preference` halves of the `type` description are updated as specified; no other type description changes.
- FR-15.3: `memory_save`'s tool description carries the scope sentence.
- FR-15.4: No runtime behavior change of any kind — description strings and guide text only; the write path gains no validation, and a persona-shaped save still succeeds.

**Acceptance criteria:**
- AC-15.1: `get_usage_guide` returns the new bullet and the reworded Rule Of Thumb under the bumped version; `npm test` passes with the synced assertion.
- AC-15.2: The registered `memory_save` tool description contains the scope sentence, and the `type` input description contains the enriched `fact`/`preference` wording.
- AC-15.3: `memory_save` accepts a persona-shaped save unchanged (guidance, not enforcement).

**Files expected to change:** `GUIDELINES.md`, `src/__tests__/reasoning-audit-tools.test.ts`, `src/schemas/memory.ts`, `src/tools/memory.ts`. Effort: **XS**.

**Considered and not done inside this WI:**
- A write-time content validator rejecting persona-shaped saves — pattern-matching on content is both easy to false-positive and over-engineered at current store scale; guidance is the proportionate tool. Revisit only if persona rows keep appearing *after* this WI ships.
- `agent_id`-scoped recall (persona rows would at least only surface for their own agent) — a real ranking-design change; belongs with the v1.4.0 blended-scoring work behind the evidence gate, not in a hygiene item.
- Cleaning the persona rows already present in any live store — operator action via `memory_delete`, not a code deliverable of this release.

## 8. Considered but NOT doing (and why)

- **WI-12 (draft 0.1) — time-decay tie-break for auto-recall: DROPPED in 0.2.** Code review found the decay slot sits after the BM25 float criterion in the comparator (HT-12a); two distinct memories essentially never tie exactly on BM25, so the decayed-importance branch would effectively never execute on real data. The feature would have passed its acceptance tests (constructed ties) while changing nothing observable for users — and it was already a flagged deviation from the owner's own v1.4.0 gating decision. Owner decision 2026-07-12: do not ship features with near-zero practical effect. The idea folds into the v1.4.0 blended-score work (where recency would be a weighted term in one formula, not a dead tie-break slot), behind the existing evidence gate, validated against WI-11's eval base.
- **WI-14 (draft 0.1) — bounded importance nudge from proven usage: DROPPED in 0.2.** Two independent reasons. (1) Same structural problem as WI-12: `importance` is itself a post-BM25 tie-break in both comparators (HT-12a), so a +1 importance change is equally invisible on real orderings. (2) Insufficient evidence: the feature's premise — that `used` feedback accumulates in meaningful volume — is exactly what the Wave-4 evidence gate (G-a: ≥20–30 used events) was designed to measure, and that measurement window has not run. Shipping a consumer of a signal before confirming the signal exists repeats the "build ahead of demand" mistake. Reconsider inside the v1.4.0 blended-score work if the gate opens. (The review also found an off-by-one in the draft's trigger design between the two feedback call sites — moot now, but recorded in the review session `sess_b92af065` for whoever picks this up later.)
- **Step-level reasoning provenance** (which memory a specific reasoning step used): this repo's own measured history (`mem_6f05457c`) shows this exact feature shape — an optional field the agent must proactively fill, consumed only by an audit/report layer with near-zero real traffic — dies at ~0% adoption regardless of documentation quality. **Rejected.**
- **Full reasoning graph / backtracking:** `reasoning_step_marks` already covers pivotal-moment flagging; no demonstrated need for true branching. **Rejected.**
- **Server-side AI auto-summarization of reasoning traces:** the server has no model access; the calling agent's own `conclusion` field already serves this purpose. **Rejected** — the idea misreads the architecture.
- **Usage-based *ranking* boost:** this is WI-6, explicitly Wave-4, gated behind the §9.3 evidence conditions, none of which can be met yet. **Not pulled forward.**
- **`memory_search` decay ranking:** doubly out — blocked by `LIMIT`/`OFFSET` pagination correctness *and* by HT-12a's tie-break futility. Belongs to the v1.4.0 blended-score design if it belongs anywhere.
- **Semantic/vector embeddings, any new dependency:** unchanged from Wave 3 — breaks the zero-dependency/local-only architecture. **Rejected.**

## 8.1. Under owner monitoring (2026-07-18) — not a work item, no target release yet

This is not a scheduled WI: no target release, no acceptance criteria, no code change. It records an owner decision so a future contributor doesn't have to re-derive it from telemetry or session transcripts.

**Observation (live-store snapshot, 2026-07-18, `MEMORY_TELEMETRY=on`, 111 `tool_usage_events` rows spanning 2026-07-16 → 2026-07-18):**
- The manual/standalone durable-memory CRUD surface — `memory_save`, `memory_search`, `memory_list`, `memory_delete` — has **zero calls**. All 19 memories currently in the store were created exclusively through `reasoning_complete_session`'s auto-save path (`save_as_memory`/`memory_mode='always'`), never via a direct `memory_save` call.
- The reasoning audit tier — `reasoning_get_trace`, `reasoning_list_sessions`, `reasoning_search_steps`, `reasoning_list_milestones`, `reasoning_get_session_outline`, `reasoning_mark_step` — also has **zero calls**; `reasoning_step_marks` has zero rows.
- By contrast, the automatic pipeline (`reasoning_start_session` auto-recall → work → `reasoning_complete_session` auto-save + `used_memory_ids` feedback) is fully exercised and works: 12 of 19 memories (63%) have received at least one `"used"` feedback event. The dividing line is *automatic/event-driven tools* (used) vs. *manual/exploratory tools requiring the agent to act outside that pipeline* (unused), not "reasoning vs. memory."

**Root cause diagnosed (same session):** not a code defect. GUIDELINES.md's conflict-check ("if two recalled memories conflict, replay the source trace") and mark-step guidance were soft suggestions with no concrete trigger, so they never fired — even though the qualifying precondition (2–3 related memories recalled at once) occurred in 18 of 25 sessions (72%). Fixed in the same session: `GUIDELINES.md` bumped `2026-07-17.v6` → `2026-07-18.v7`, turning both into explicit, concrete steps (Moment 1 conflict IF/THEN; Moment 2 mark-step trigger list), with the `guide_version`/`guidance_version` test assertions synced in `src/__tests__/reasoning-audit-tools.test.ts`.

**Owner decision (2026-07-18):** continue using the MCP as-is with the tightened guide to collect more real usage data. If usage of the manual CRUD surface and/or the reasoning audit tier stays at or near zero after this observation window, the owner intends to remove those tools in a future release rather than keep unused surface area. No target version is assigned until the window closes and the decision is made — this deliberately stays out of the v1.3.5 WI list above.

**How to apply if you pick this up later:** re-run the `tool_usage_events` query grouped by `tool_name` (plus a `reasoning_step_marks` row count) against `~/.memory-mcp-server/memory.db` to see whether the guide fix moved the needle. If it did not, a removal spec should follow this document's own evidence-first format (§8 precedent) — but treat it as a breaking API-surface change requiring its own `MCP_VERSION`/`package.json` bump and `CHANGELOG.md` entry, unlike WI-15's docs-only change.
